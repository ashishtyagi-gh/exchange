🎯Design Decisions

DATA STRUCTURE
Price levels: sorted map (red-black tree)
A sorted map for bids (descending order — best bid first) and a sorted map for asks (ascending — best ask first). O(log N) insert, remove, and lookup. All tree nodes are allocated from the per-market memory pool — no global heap. At N = 100 price levels, the entire working set is ~6 KB per side, staying in L1/L2 cache.
Each price-level lookup traverses ~7 tree nodes (log₂ 100 ≈ 7). Each step may be a separate cache line. This is the primary bottleneck — see throughput section. A flat sorted array would win on lookup but loses on N-level sweeps (O(N) shifts). Benchmarks confirmed the tree is optimal for our workload profile.
DATA STRUCTURE
Per-level order queue: FIFO queue with lazy deletion
Each price level holds a FIFO queue of order IDs. Matching consumes from the front. Cancel is O(1): mark the order as Canceled in the order lookup table. The queue entry is silently skipped the next time that level is matched through. Partially-filled makers are pushed back to the front of the queue to preserve their original time priority — their sequence number never changes.
Deep OTM levels accumulate dead entries that are never reaped. In production, an intrusive doubly-linked list would allow O(1) physical removal. Documented explicitly as a known limitation.
CORRECTNESS
Determinism: monotonic seq_ counter, no wall clocks
Every command processed increments a per-market sequence counter. This counter is the sole source of time ordering — no wall clocks, no system timestamps, no thread IDs. Consequence: given the same sequence of place and cancel inputs in the same order, the engine produces byte-for-byte identical fills and book states every run, on any machine.
Wall-clock timestamps are useful for regulatory reporting and latency measurement but are not needed for correctness. They are trivially addable to events as a separate field without touching the matching logic.
CORRECTNESS
Self-trade prevention: Cancel Newest (incoming order)
When the incoming order’s account matches the resting maker’s account, the incoming order is immediately rejected (reason: SelfTrade). The resting maker is put back at the front of the queue with its original queue position intact. Three policies were considered: Cancel Newest (chosen), Cancel Oldest, Cancel Both. Cancel Newest was chosen because: (1) the maker's priority is preserved — least disruptive to the book, (2) the maker already has a confirmed resting acknowledgement, (3) simplest to audit.
Cancel Oldest (removing the maker) might be preferred in venues that want to prevent quote stuffing. Cancel Both simplifies state but destroys resting liquidity. Policy is a configuration choice in real venues.
MEMORY
Per-market memory arena: pre-allocated slab → usage tracker → pool recycler → data structures
Each market gets a private 16 MB buffer pre-allocated at startup. Allocation passes through four layers: raw buffer → bump-pointer allocator → usage tracker → pool recycler → data structures. The pool recycler hands back previously freed order slots and tree-node slots. After an order is canceled and a new one is placed, committed memory stays flat at steady state — no OS memory calls on the hot path. New placements are rejected at the 70% committed watermark. Cancels always proceed (they free memory). The pool is declared last, so it is destroyed first — ensuring the cleanup sequence is safe.
Allocations larger than 1024 bytes (e.g. hash table bucket array resizes) bypass the pool recycler and count against the committed watermark permanently. The 30% safety reserve is sized to accommodate this.
CORRECTNESS
Partial fill priority preservation: maker pushed to front, seq unchanged
When a resting maker is partially filled, it is pushed back to the front of the price level’s FIFO queue — not the back. Its original sequence number is never modified. This is the correct behaviour for price-time priority: the maker arrived earlier than any new order at this price, so it must remain first in queue for the next fill. A common mistake is to re-enqueue at the back (re-prioritising as if the order had just arrived), which violates time priority.

The same logic applies to self-trade prevention: when an incoming order is rejected for STP, the resting maker is also pushed back to the front before returning — its position in the queue is exactly as if the STP check had never run.
Pushing to the front of the queue is O(1). The alternative (intrusive doubly-linked list) would also be O(1) but adds complexity. The queue is sufficient here and is already allocated from the per-market memory pool.

2Part 2 — Multiple Markets & Concurrency

1 Thread per Market (1:1 model — high-volume instruments)

  • Each consumer thread owns exactly one market
  • Commands arrive via a private lock-free single-producer/single-consumer (SPSC) queue (4096 slots)
  • Consumer calls place or cancel on the book — no locking needed
  • On stop(): drains remaining queue before exiting (no lost commands)
  • Thread pinnable via isolcpus + numactl for NUMA-local allocation
  • In production, a low-volume market (e.g. an illiquid symbol) need not get a dedicated core — multiple markets can be multiplexed onto one thread by round-robin polling their queues in a single loop, trading latency for core efficiency

Ordering Guarantees

  • Within a market: total order — every event has a strictly increasing sequence number
  • Across markets: no ordering guarantee — each market has an independent seq counter starting at 1
  • Event streams are per-market and independently replayable
  • A downstream log consumer gets a totally ordered stream per market
  • Cross-market atomic snapshots require an external coordinator (not implemented — out of scope)

Adding Markets

  • Construct a new memory pool, order book, and input queue for the market
  • Wrap them in a market handle and pass to a new consumer thread
  • Start the consumer — fully independent from all existing markets
  • Linear throughput scaling: N markets = N cores, no shared state

Why Not a Thread Pool?

  • A pool requires routing logic — which thread handles which market?
  • Any cross-thread handoff adds cache-line coherence traffic
  • The book is NOT thread-safe by design — concurrent access is the bug
  • 1:1 is simpler, saturates a full core, and is the dominant model in production exchanges (e.g. CME Globex, Nasdaq ITCH)

Primary Bottleneck

  • Sorted-map cache pressure: ~7 pointer-chased tree nodes per price-level lookup
  • At N=100 levels, working set ≈ 6 KB/side — fits in L1/L2, but pointer-chasing still dominates
  • Each node traversal is a potential cache-line miss after any modification
  • This is the irreducible cost of a comparison-sorted tree — not a bug

Secondary Bottleneck

  • Order table growth: lazy deletion leaves Filled entries in the order lookup table
  • After 500K iterations the table has 500K stale entries — lookup degrades slowly
  • Production fix: periodic reclaim pass or intrusive doubly-linked list for O(1) physical removal
  • The near-spread benchmark degrades over its 500K iteration run; the price-drift benchmark reuses fewer orders so grows slower

⚖Tradeoffs & Production Considerations

What I choseWhy (time / correctness)What I'd change for production
Lazy cancel — dead entries accumulate in deque O(1) cancel, no data structure changes. Correct: dead entries are skipped during matching. Intrusive doubly-linked list per price level: O(1) physical removal at cancel, no garbage accumulation.
No book query API (top-of-book, depth, order status) The exercise focuses on matching correctness. Adding query is straightforward. Expose top_of_book(), depth(N), order_status(id). Reads are safe on the consumer thread; snapshots for external callers need a copy or MVCC.
No persistence / recovery Out of scope per the problem statement. Append-only command log (WAL). On crash, replay the log to reconstruct book state. The seq counter provides a natural checkpoint cursor.
Self-trade: Cancel Newest Preserves maker priority. Deterministic. Simplest to audit. Make it a per-venue configuration. Some venues prefer Cancel Both to punish wash-trade attempts.
Input queue: fixed 4096-slot ring Zero-allocation, cache-friendly. Sufficient for benchmark workloads. Dynamic capacity or a two-level queue (fast ring + overflow heap queue) for burst traffic handling.
Order lookup table grows unbounded Simplest correct implementation. Lazy deletion keeps the hot path O(1). Slab-per-session: at end of trading day, reset the entire arena in one shot — no per-order cleanup needed. Or, on the consumer thread itself (not a background thread — the table is not thread-safe), sweep stale entries during idle periods between bursts.
No auth, collateral, risk checks Explicitly excluded by problem statement. Pre-trade risk gate sits in front of the input queue, not inside the engine. Engine stays pure and deterministic.

🌐Part 3 — Toward a Distributed Exchange (design only)

Two regions: Illinois and Tokyo. Each market is owned by exactly one region. A BTC order placed in New York must reach the Tokyo engine to match.

ROUTING & ACKNOWLEDGEMENT
What does a New York client see and when?
The NY gateway accepts the order, assigns a local client-visible receipt (timestamp + gateway seq), and immediately returns a received acknowledgement to the client (~0.2 ms for a co-located client; ~0.5–2 ms for a retail client in the same city). This is NOT a fill confirmation — it is a delivery guarantee. The order then travels to Tokyo (~130 ms one-way on a leased trans-Pacific line). The Tokyo matching engine processes it, emits Fill/Ack/Reject events, and streams them back to NY. The client receives the true match acknowledgement at 2 × one-way + engine latency (~260 ms floor from NY).

Clients that care about latency place orders directly via the Tokyo gateway (access fee trade-off). The NY gateway is for regional clients who accept the latency floor.
The 260 ms floor is determined by the speed of light, not software. You cannot improve it without moving the matching engine closer to the client.
DETERMINISM & REPLAY
WAL + seq numbers make state reconstructable
Every command the Tokyo engine accepts is written to an append-only WAL (write-ahead log) before being dispatched to the matching engine. The WAL entry contains: global seq, command payload, and a hash of the previous entry (chain integrity).

After a crash: replay the WAL from the last checkpoint. The matching engine's seq counter provides a natural cursor. Since the engine is deterministic (no wall clocks, pure function of inputs), replay produces exactly the same book state.

A replica region (Illinois) subscribes to the WAL stream. It maintains a shadow book that stays ~130 ms behind. This replica can serve read-only queries (market data, order status) without touching the primary.
The WAL is the serialization point. Its write throughput is the new ceiling. In production this would be a high-throughput sequential append log (Chronicle Queue, LMAX Disruptor-style ring). NVMe sync latency is ~20–100µs — well under the 130 ms network floor.
FAILURE MODES
Link degradation and regional failover
Link degradation (partial): The NY gateway buffers commands locally. Orders accumulate in a durable queue (bounded — e.g. 1M slots; beyond that, new orders are rejected with a "gateway congested" response rather than silently dropped or accepted with a false promise). Once the link recovers, they are replayed in arrival order. Clients receive a "gateway buffering" notice. No data loss; latency increases.

Tokyo goes down: The Illinois replica has a consistent but stale book (~130 ms behind). It cannot safely take over without risk of duplicate fills or missed cancels for in-flight orders. Safe path: pause all new order acceptance, drain in-flight orders from the WAL up to the last confirmed seq, then promote Illinois as primary. This takes seconds, not minutes.

Risk of premature takeover: If Illinois promotes itself while there are unconfirmed orders in the NY→Tokyo pipeline, those orders will be re-submitted and may match again, causing double fills. This is the critical window. Mitigation: NY gateway tracks the highest seq it has received a Fill/Ack/Reject for. Failover is safe only after Illinois's WAL position reaches that seq.

Re-open auction after failover: Even after a clean seq-safe promotion, participants' local views of the book may have diverged during the outage window — some clients received fills, others received timeouts, and resting orders may have been acknowledged by Tokyo but not yet visible to Illinois clients. Rather than resuming continuous matching into a potentially inconsistent state, the promoted Illinois engine should:
  1. Start with a fresh order book — all pre-failover resting orders are canceled with a CancelAck event
  2. Open a call auction window (e.g. 60 seconds) — participants submit new or revised orders; no matching occurs yet
  3. At auction close, match all eligible orders at a single clearing price that maximises volume
  4. Resume continuous matching from that clean, agreed-upon book state
This is the standard exchange recovery procedure (used by NYSE, LSE, and most CME products after a halt). It re-establishes a coherent book for all participants simultaneously rather than inheriting ambiguous pre-failover state.

Regulatory dimension: In most jurisdictions (SEC, ESMA, MAS), an unplanned engine outage is a reportable market disruption event. The exchange must:
  • Notify the regulator within a defined window (e.g. 15 minutes under MiFID II Art. 12)
  • Preserve the full WAL and event log covering the outage period — this is the audit trail
  • Conduct the re-open auction rather than resuming silently — regulators require a visible, time-stamped halt/resume boundary
  • Publish a post-incident report explaining root cause, affected instruments, and any erroneous trades
Skipping the auction and resuming into a stale book is not just technically risky — it is likely a regulatory violation.
You cannot eliminate the failover window with a two-region design. Three regions with Raft/Paxos consensus would allow safe automatic failover — but only if the three nodes are co-located in the same metro (e.g. three data centres in the Chicago area, ~0.3–2 ms inter-node RTT), making each consensus round-trip ~1–4 ms per order. If the quorum nodes were spread across Illinois, Tokyo, and London, every order would pay the slowest quorum leg (~140 ms to London), which is unusable. Most exchanges keep the consensus cluster local and treat the remote region as an async follower, accepting the manual failover window rather than the latency tax of a geographically distributed quorum.
DESIGN CHANGES
What would change in Parts 1 & 2 if I had known from the start?
1. WAL-first command dispatch: Instead of pushing directly to the SPSC queue, the Consumer would write to the WAL first, then dispatch. This adds ~20–100µs latency per command but makes the system crash-safe by design.

2. Global order IDs: Order IDs would be gateway-scoped (region prefix + local monotonic ID) rather than client-assigned. This prevents ID collisions when orders from multiple regions converge on one engine.
The core matching engine (OrderBook, PriceLevel, SPSC, arena) is unchanged — it is already pure and deterministic. The distributed concerns are a wrapper around it, not inside it. This is the correct abstraction boundary.
